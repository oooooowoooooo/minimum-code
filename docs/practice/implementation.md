# Implementation

## Overview

This chapter covers building the knowledge base application. It is organized by implementation phase, with both Python (backend) and TypeScript (frontend) code.

Every code example in this chapter was designed to be generated by AI with the right prompts. The goal is not to memorize the code -- it is to understand the structure well enough to direct AI effectively.

## Phase 2: Data Layer

### Database Models (Python)

```
Generate SQLAlchemy models for the knowledge base application.

Models needed:
1. User - id (UUID), email (unique), name, hashed_password, role (enum: admin/member),
   created_at, updated_at, is_active
2. Document - id (UUID), user_id (FK), title, filename, file_type (enum: pdf/md/txt),
   file_size, status (enum: uploading/processing/ready/failed), chunk_count,
   created_at, updated_at
3. DocumentChunk - id (UUID), document_id (FK), chunk_index, content (Text),
   embedding (vector(1536)), metadata (JSON), created_at
4. Conversation - id (UUID), user_id (FK), title, created_at, updated_at
5. Message - id (UUID), conversation_id (FK), role (enum: user/assistant),
   content (Text), citations (JSON), created_at

Use:
- SQLAlchemy 2.0 declarative style with Mapped type annotations
- async engine with asyncpg
- pgvector for the embedding column
- UUID primary keys
- Proper indexes on foreign keys and frequently queried columns
- Cascade deletes (deleting a document deletes its chunks)
```

### Pydantic Schemas

```
Generate Pydantic schemas for API request/response validation.

Schemas needed:
1. UserCreate, UserLogin, UserResponse, TokenResponse
2. DocumentCreate, DocumentResponse, DocumentListResponse
3. QueryRequest, QueryResponse, CitationSchema
4. ConversationResponse, MessageResponse
5. PaginationParams (page, per_page)

Rules:
- Use Pydantic v2 with ConfigDict
- Separate input schemas (what the client sends) from output schemas (what the API returns)
- Input schemas should have validation (email format, string length limits, etc.)
- Output schemas should use from_attributes=True for ORM model conversion
- All datetime fields should be serialized as ISO 8601
```

## Phase 3: Core Services

### Document Service

```
Implement the DocumentService class with these methods:

1. upload_document(user_id: UUID, file: UploadFile) -> Document
   - Validate file type and size
   - Save file to disk (UPLOAD_DIR / user_id / document_id / filename)
   - Create Document record with status "uploading"
   - Trigger async processing (call process_document)
   - Return the document

2. process_document(document_id: UUID) -> None
   - Update status to "processing"
   - Extract text from file based on file_type
   - Chunk the text
   - Generate embeddings for chunks
   - Store chunks in database
   - Update status to "ready" and chunk_count
   - On any error: update status to "failed", log error

3. get_document(user_id: UUID, document_id: UUID) -> Document
   - Fetch document, verify ownership
   - Raise NotFoundError if not found, ForbiddenError if not owner

4. list_documents(user_id: UUID, page: int, per_page: int) -> DocumentListResponse
   - Paginated list of user's documents
   - Sort by created_at descending

5. delete_document(user_id: UUID, document_id: UUID) -> None
   - Verify ownership
   - Delete file from disk
   - Delete document (cascades to chunks)
```

### Query Service

```
Implement the QueryService class with these methods:

1. query(user_id: UUID, question: str, conversation_id: UUID | None) -> QueryResponse
   - Generate embedding for the question
   - Search for relevant chunks (semantic search)
   - Construct prompt with context chunks
   - Call LLM to generate answer
   - Format response with citations
   - Store question and answer in conversation
   - Return QueryResponse

2. search_chunks(query_embedding: list[float], user_id: UUID, limit: int = 5) -> list[ChunkResult]
   - Use pgvector cosine similarity search
   - Only search chunks from the user's documents
   - Return top N most relevant chunks with scores

3. build_prompt(question: str, chunks: list[ChunkResult]) -> str
   - Construct a prompt that includes:
     - System instructions (be accurate, cite sources, say "I don't know" if unsure)
     - The relevant document chunks as context
     - The user's question
   - Format clearly for the LLM

4. generate_answer(prompt: str) -> str
   - Call the LLM API
   - Handle rate limits and errors
   - Return the generated text

5. extract_citations(answer: str, chunks: list[ChunkResult]) -> list[Citation]
   - Parse the answer to identify which chunks were referenced
   - Return citation objects with document_id, chunk_text, relevance_score
```

### User Service

```
Implement the UserService class with these methods:

1. register(email: str, password: str, name: str) -> User
   - Check if email already exists
   - Hash password with bcrypt
   - Create user record
   - Return user (without password hash)

2. login(email: str, password: str) -> TokenResponse
   - Find user by email
   - Verify password with bcrypt
   - Generate JWT access token (15 min) and refresh token (7 days)
   - Return tokens

3. get_current_user(token: str) -> User
   - Decode and validate JWT
   - Fetch user from database
   - Verify user is active
   - Return user

4. refresh_token(refresh_token: str) -> TokenResponse
   - Validate refresh token
   - Generate new access token
   - Return new token pair
```

## Phase 4: Document Processing Pipeline

### Text Extraction

```
Implement text extraction functions for different file types:

1. extract_text_from_pdf(file_path: Path) -> list[PageContent]
   - Use PyMuPDF (fitz) to extract text
   - Return list of pages with text content and page numbers
   - Handle scanned PDFs (raise error -- no OCR for now)

2. extract_text_from_markdown(file_path: Path) -> str
   - Read the markdown file
   - Strip markdown formatting for embedding (keep raw text)
   - Preserve structure metadata (headers, sections)

3. extract_text_from_text(file_path: Path) -> str
   - Read plain text file
   - Handle encoding detection (UTF-8 default, fallback to latin-1)

Error handling:
- File not found: raise DocumentNotFoundError
- Corrupt file: raise DocumentProcessingError with details
- Empty file: raise DocumentProcessingError("File is empty")
```

### Text Chunking

```
Implement a text chunking function:

def chunk_text(
    text: str,
    chunk_size: int = 500,      # target chunk size in tokens
    chunk_overlap: int = 50,    # overlap between chunks
    separators: list[str] = None  # split points in priority order
) -> list[TextChunk]:

Algorithm:
1. Start with the full text
2. Try to split at the first separator (e.g., paragraph break)
3. If the resulting chunk is within chunk_size, keep it
4. If too large, try the next separator (e.g., sentence break)
5. If still too large, split at word boundary
6. Maintain overlap: the last chunk_overlap tokens of each chunk
   appear at the start of the next chunk

Each TextChunk should contain:
- content: str (the text)
- start_offset: int (position in original text)
- end_offset: int (position in original text)
- metadata: dict (page number if available, section header if available)

Write unit tests for:
- Normal text with clear paragraph breaks
- Very long paragraphs (no natural breaks)
- Text shorter than one chunk
- Empty text
- Text with special characters and unicode
```

        """Generate embedding for a single text."""

## Summary

Implementation is where the architecture becomes real. The key principles:

1. **Build incrementally.** Get one piece working before starting the next.
2. **Use AI for generation, not for decisions.** You decide what to build. AI helps you build it.
3. **Test as you go.** Do not wait until the end to test.
4. **Refactor when you see mess.** Clean code is easier to extend.
5. **Document decisions, not code.** Code should be self-documenting. Document the "why."
